## F-06 App State Machine `[TODO]`

`app/app.ts` — owns all application state. No other module mutates state directly. The map module and UI module are called with data; they return nothing (side-effects only). State transitions are the only place storage reads and writes occur.

### Module interface — exported symbols

```typescript
export type AppState =
  | { mode: "loading" }
  | { mode: "error"; message: string }
  | { mode: "browsing"; userLat: number | null; userLng: number | null; allSigns: Sign[]; activeSigns: Sign[] }
  | { mode: "parked";   spot: SavedSpot; allSigns: Sign[]; nearbySigns: Sign[] }

export type App = {
  getState(): AppState
  onSaveSpot(spot: SavedSpot): void
  onClearSpot(): void
  onHereNow(): void
  /** Called by main.ts via setInterval every 60 s. No setInterval inside app.ts. */
  tick(now: Date): void
}

/**
 * Creates the app state machine. main.ts calls this after initMap(), then
 * calls setInterval(() => app.tick(new Date()), 60_000) — the interval lives
 * in main.ts, not here, so app.ts has no side effects.
 */
export function createApp(
  deps: { storage: SpotStorage; renderState: (state: AppState) => void },
  initialData: { signs: Sign[]; fetchTime: Date }
): App
```

### F-06.2 Startup Data Load

At startup: fetch `data/latest.json` and `data/street-cleaning.json` in parallel using `Promise.all`. Run `filterLoadTimeNoise(signs, now)` on the sign data, then check storage for a saved spot. Transition to `"browsing"` if no spot exists, `"parked"` if a spot exists. Street cleaning entries are stored in a module-level variable accessible to map interaction handlers. If `street-cleaning.json` fails to load, the app continues normally with an empty entries list — the street popup simply shows "No cleaning schedule found" rather than crashing.

**Tests**
- GIVEN `latest.json` loads successfully and storage is empty, WHEN startup completes, THEN state is `"browsing"` and `allSigns.length` is less than 116 (noise filtered).
- GIVEN `latest.json` loads successfully and a saved spot is in storage, WHEN startup completes, THEN `getState().mode === "parked"` and `getState().nearbySigns` is a `Sign[]`.
- GIVEN `latest.json` fails to load (network error), WHEN startup fails, THEN `getState().mode === "error"` and `getState().message` is a non-empty string.
- GIVEN `street-cleaning.json` fails to load, WHEN startup completes, THEN the app still reaches `"browsing"` or `"parked"` state (graceful degradation — cleaning data is non-critical).

### F-06.3 State Transitions

**Tests**
- GIVEN state is `"browsing"`, WHEN SAVE MY SPOT is confirmed with a valid `SavedSpot`, THEN state transitions to `"parked"`.
- GIVEN state is `"parked"`, WHEN CLEAR MY SPOT is tapped, THEN state transitions to `"browsing"` and the spot is removed from storage.
- GIVEN state is `"parked"`, WHEN I'M HERE NOW is tapped, THEN state remains `"parked"` (not a state change — triggers a map center operation only).

### F-06.4 60-Second Tick

On each tick (driven by `app.tick(now)` called from main.ts every 60 seconds): re-run `filterActive(allSigns, now)` and, if in parked mode, re-run `filterNearby(allSigns, spot.lat, spot.lng, 150, now)`. Push updated state to `renderState`.

**Tests**
- GIVEN a sign whose window closes between ticks, WHEN `tick(now)` is called with a `now` after the sign's `end_iso`, THEN `getState().activeSigns` does not contain that sign.
- GIVEN no sign status changes between ticks, WHEN `tick(now)` is called, THEN `renderState` is called exactly once with the current state and does not throw.
- GIVEN the app is in parked mode and a sign becomes active within 150 m, WHEN `tick(now)` is called with `now` inside that sign's window, THEN `getState().nearbySigns` includes that sign.
