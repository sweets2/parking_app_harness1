### F-02.1 SIGN_REASONS and SignReason

```typescript
export const SIGN_REASONS = ["CONSTRUCTION", "MOVING", "EVENT", "DELIVERY"] as const;
export type SignReason = (typeof SIGN_REASONS)[number];
```

`as const` makes `SIGN_REASONS` a readonly tuple so `SignReason` is derived from it, not duplicated. Adding a new reason type in one place propagates everywhere automatically.

**Tests** (compiler-level â€” verified by `npm run typecheck`)
- GIVEN `SignReason`, THEN it is the union `"CONSTRUCTION" | "MOVING" | "EVENT" | "DELIVERY"`.
- GIVEN a variable typed as `SignReason`, THEN assigning `"FILM"` to it must produce a TypeScript error.
- GIVEN `SIGN_REASONS`, THEN it is usable as an iterable array at runtime (for validation loops in the fetcher).

### F-02.2 Sign Interface

```typescript
export interface Sign {
  id:              string;
  address:         string;
  reason:          SignReason;
  permit_number:   string;
  lat:             number;
  lng:             number;
  start_date:      string;   // original M/D/YYYY from API â€” preserved for display only
  start_time:      string;   // original HH:MM:SS from API â€” preserved for display only
  stop_date:       string;   // original M/D/YYYY from API â€” preserved for display only
  end_time:        string;   // original HH:MM:SS from API â€” preserved for display only
  start_iso:       string;   // ISO 8601 local datetime â€” use this for all time logic
  end_iso:         string;   // ISO 8601 local datetime â€” use this for all time logic
  active_at_fetch: boolean;
}
```

`lat`/`lng` are the transformed versions of the API's `latitude`/`longitude`. `start_iso`/`end_iso` are derived from the raw date strings by the fetcher. The raw strings are kept on the object for display purposes only â€” never use them for comparisons.

**Tests** (compiler-level)
- GIVEN a `Sign` object, THEN `sign.lat` and `sign.lng` are typed as `number` (not optional).
- GIVEN a `Sign` object, THEN `sign.reason` is typed as `SignReason`, not `string`.
- GIVEN an object with `latitude: number` instead of `lat: number`, THEN assigning it to a `Sign` variable must produce a TypeScript error (field name mismatch between raw API and internal type).

### F-02.3 ParkingData Interface

```typescript
export interface ParkingData {
  fetched_at: string;   // ISO 8601 UTC â€” when the fetch ran
  count:      number;   // must equal signs.length
  signs:      Sign[];
}
```

This is the shape of `data/latest.json`. The PWA fetches this file and reads from it directly.

**Tests** (compiler-level)
- GIVEN a `ParkingData` object, THEN `signs` is typed as `Sign[]`, not `any[]`.
- GIVEN code that accesses `data.fetched_at`, THEN `fetched_at` typed as `string` (the ISO value is validated at runtime, not at compile time).

### F-02.4 RawSign and RawApiResponse Interfaces (fetcher-only)

```typescript
export interface RawSign {
  id:           string;
  address:      string;
  reason:       string;       // validated against SIGN_REASONS before use; typed as string here
  permit_number: string;
  latitude:     number;       // renamed to lat in Sign
  longitude:    number;       // renamed to lng in Sign
  start_date:   string;
  start_time:   string;
  stop_date:    string;
  end_time:     string;
}

export interface RawApiResponse {
  status: string;
  data:   RawSign[];
}
```

These types are only imported by `fetcher/fetch.ts`. The PWA must never import them.

**Tests** (compiler-level)
- GIVEN `RawSign`, THEN it has `latitude` and `longitude` (not `lat`/`lng`).
- GIVEN a `RawSign` value, THEN assigning it to a `Sign` variable must produce a TypeScript error (incompatible shapes).
