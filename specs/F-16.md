## F-16 Next Violation Window `[TODO]`

`nextViolationWindow(signs: Sign[], spot: SavedSpot, now: Date): ViolationWindow | null` in `shared/parking-logic.ts`. Returns the nearest upcoming or currently-active sign conflict for the saved spot, or `null` if none. Pure function, injectable `now`.

```typescript
type ViolationWindow = {
  sign: Sign;
  minutesUntilActive: number;  // 0 if already active, positive if upcoming
}
```

**Tests**
- GIVEN no signs are within 150 m of the spot, WHEN `nextViolationWindow` is called, THEN returns `null`.
- GIVEN one currently-active sign within 150 m, WHEN called, THEN returns `{ sign, minutesUntilActive: 0 }`.
- GIVEN one sign starting in 90 minutes within 150 m, WHEN called, THEN returns `{ sign, minutesUntilActive: 90 }`.
- GIVEN two signs starting in 90 and 45 minutes, WHEN called, THEN returns the sign with `minutesUntilActive: 45` (soonest).
- GIVEN one active sign and one upcoming sign, WHEN called, THEN returns the active sign with `minutesUntilActive: 0`.
- GIVEN all nearby signs have already expired, WHEN called, THEN returns `null`.
- GIVEN a sign that is nearby but not yet started (start is in 3 days), WHEN called, THEN returns that sign with the correct `minutesUntilActive`.
- GIVEN an empty `signs` array, WHEN called, THEN returns `null` without throwing.

**Known data limitation:** The Hoboken API only returns signs that have already been posted (not future ones not yet in the system). A sign posted overnight will not appear until after the next 6 AM fetch. `nextViolationWindow` cannot warn about signs not yet in `latest.json`. This is documented behavior, not a bug.
