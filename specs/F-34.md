## F-34 Street Violation Highlights (Cleaning-Only)

Semi-transparent shading layer: red for every block with an active street-cleaning
restriction right now, orange for blocks that will be in violation within the
next 60 minutes. A legend toggle (same layout as the Tow Zones legend) lets users
hide the layer. Highlights update automatically at each :00 hour boundary.

Tow zones already have their own distinct visual (red/white casing polylines).
The shading layer covers **street cleaning only** — adding tow signs flooded the
map with ~16 permanent permit signs painting nearly every street red at all times.

> **F-45 extension:** Block-level clipping was added in F-45. `renderViolationHighlights`
> keys by `(street, side, location)` so entries with different location ranges on the
> same street are rendered as independent segments, not merged into a single full-street
> highlight.

- `app/main.ts` — `scheduleViolationRefresh`, violation re-renders at sign state transitions, legend toggle wiring, initial render + schedule after `createApp()`

### What "violation" means

Street-cleaning entries from `data/street-cleaning.json` are checked per refresh:

- `isScheduleActiveNow(entry.schedule, now)` → active (red)
- `isScheduleUpcomingSoon(entry.schedule, now)` → upcoming within 60 min (orange)

**Active takes priority per block**: for a given `(street, side, location)` key, active
beats upcoming. Two different location ranges on the same street+side each carry their
own priority independently.

**Street name normalisation** — `normalizeToGeometryKey(entry.street)` maps cleaning
entry street names to road-geometry keys:

- Uppercases, strips trailing periods (cleaning data uses "Adams St.", geo has "ADAMS ST")
- Expands abbreviations: STREET→ST, AVENUE→AVE, HIGHWAY→HWY, BOULEVARD→BLVD, PLACE→PL, COURT→CT, DRIVE→DR, ROAD→RD, TERRACE→TER
- Converts spelled-out ordinals to numeric: FIRST→1ST, SECOND→2ND, … SIXTEENTH→16TH
  (cleaning data uses "Eighth St.", geo has "8TH ST")

### `app/map.ts` — modifications

**New module-level constants and state** (add after `_roadGeometry`):

```typescript
const ORDINAL_TO_NUMERIC: Record<string, string> = {
  FIRST: "1ST", SECOND: "2ND", THIRD: "3RD", FOURTH: "4TH",
  FIFTH: "5TH", SIXTH: "6TH", SEVENTH: "7TH", EIGHTH: "8TH",
  NINTH: "9TH", TENTH: "10TH", ELEVENTH: "11TH", TWELFTH: "12TH",
  THIRTEENTH: "13TH", FOURTEENTH: "14TH", FIFTEENTH: "15TH", SIXTEENTH: "16TH",
};

let _violationLayers: LeafletLayer[] = [];
let _violationHighlightsVisible = true;
```

**Private helper `normalizeToGeometryKey`** — see signature above.

**`drawWaysHighlight(ways, color, opacity, side)`** — takes a pre-resolved ways array,
calls `mergeWays` on it, computes lane width / offset at current zoom, creates polylines,
pushes to `_violationLayers`, and conditionally adds to map. Side-aware: East/West/North/
South render on the curb-offset lane; null (or "Both") renders full-width centered.

**`drawStreetHighlight(street, color, opacity, side)`** — thin wrapper that looks up
`_roadGeometry[street]` and delegates to `drawWaysHighlight`. Used internally when no
location filtering is needed.

**`renderViolationHighlights(cleaningEntries, now)`** — groups entries by a composite
key `${normalizedStreet}|||${side}|||${location}`. Active beats upcoming per key.
For each keyed entry, calls `parseLocationLatBounds` (F-45) to restrict which road-
geometry ways are highlighted; falls back to the full street when the location can't
be parsed. Calls `drawWaysHighlight` on the filtered (or full) ways set.

---

### Side-level splitting

The `side` field in `StreetCleaningEntry` is `"East" | "West" | "North" | "South" | "Both"`.

Each unique `(street, side, location)` triple is rendered independently:
- East/West/North/South entries → `drawWaysHighlight` with a curb offset in the
  appropriate direction (same `offsetPolylinePoints` logic used by tow segments)
- "Both" or missing side → `drawWaysHighlight` with `side = null` → full-width
  centerline highlight

This prevents East and West cleaning schedules from being merged into a single
full-street color even when they carry different statuses.

### `app/main.ts` — modifications

All `renderViolationHighlights` calls take `(cleaningEntries, now)` — no signs argument.

`scheduleViolationRefresh` fires at each :00 hour boundary and calls
`renderViolationHighlights(cleaningEntries, new Date())`.

`_lastCleaningEntries` / `_lastNowForHighlights` module state is saved at the top of
`renderViolationHighlights` so zoom-change redraws can re-call with the same inputs.

### `tests/unit/map.test.ts` — test cases (12 total, inside `describe("F-34 violation highlights")`)

Fixtures:
- `ACTIVE_CLEANING` / `UPCOMING_CLEANING` — West-side Bloomfield St., non-overlapping schedules
- `EAST_ACTIVE` / `WEST_UPCOMING` / `WEST_ACTIVE` — side variants of the above

1. Active cleaning schedule → red polylines on map
2. Upcoming cleaning schedule → orange polylines
3. Active and upcoming cleaning on same `(street, side, location)` key → only red (active wins)
4. `setViolationHighlightsVisible(false)` → all violation layers removed from map
5. `setViolationHighlightsVisible(true)` → layers added back
6. `clearViolationHighlights` → all violation polylines removed from map
7. Empty road geometry → no error thrown, no polyline layers added
8. Two connected ways → one merged polyline (mergeWays)
9. Two disconnected ways → two separate polylines
10. East=active + West=upcoming → two offset polylines (red + orange)
11. East=active + West=active → two offset red polylines (no full-width)
12. East=active only → one offset red polyline (West not rendered)
