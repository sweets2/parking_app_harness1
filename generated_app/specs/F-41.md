# F-41 — Fix: Tow-Zone Curb Offset via Address Parity

## What this is

This corrects a rendering defect introduced in F-25 and F-35. No new user-visible behavior is added; existing red/orange tow-zone polylines simply appear at the curb instead of the centerline.

**Root cause:** `offsetPolylinePoints` determines which curb to offset toward by computing a dot product of (sign GPS position → road segment perpendicular). The permit API geocodes sign addresses to road centerlines, so the displacement vector is ≈ 0 and `Math.abs(dot) < 1e-9` causes the function to return the polyline unchanged. The lines render on the centerline, not the curb.

**Fix:** Pre-build `data/street-parity.json`, a `Record<string, 1 | -1>` mapping each Hoboken street name (normalized) to which perpendicular direction holds odd-numbered addresses. At render time, extract the house number from `sign.address`, look up the direction, and pass it as `forcedDir` to `offsetPolylinePoints`, bypassing the broken GPS path.

## Module contract

### `fetcher/build-street-parity.ts` (new)

Executable script. When run as main (guarded by `isMain` check), it:
1. Reads `data/road-geometry.json` for centerline geometry.
2. Queries Overpass API for all address nodes in Hoboken's bounding box.
3. For each node, finds the nearest road segment, computes the perpendicular dot product, and votes: odd house number + dot > 0 → vote +1; odd + dot < 0 → vote −1; even flips sign.
4. Emits parity (`1 | -1`) only for streets with ≥ 3 net votes.
5. Writes `data/street-parity.json` as a JSON object.

`normalizeStreet(s: string): string` — normalizes OSM `addr:street` values to keys that match `normalizeToGeometryKey` output in `app/map.ts`. Must be a verbatim copy of that function's transforms (no directionals). Keys in `street-parity.json` must match what `normalizeToGeometryKey(streetName)` produces at runtime.

### `app/map.ts` (modified)

**New module state:**
```typescript
let _streetParity: Record<string, 1 | -1> = {};
```
Reset to `{}` in the `initMap()` reset block.

**New export:**
```typescript
export function initStreetParity(data: Record<string, 1 | -1>): void
```
Sets `_streetParity = data`.

**`offsetPolylinePoints` signature change** — add optional 5th parameter:
```typescript
export function offsetPolylinePoints(
  pts: [number, number][],
  signLat: number,
  signLng: number,
  offsetM: number,
  forcedDir?: 1 | -1
): [number, number][]
```
When `forcedDir !== undefined`, skip the midpoint/dot-product block entirely and use `forcedDir` as `dir`. All existing 4-arg call sites are unaffected (TypeScript optional parameter).

**`renderTowSegments` and `renderUpcomingTowSegments`** — after computing `waypoints` from geometry but before calling `offsetPolylinePoints`, add:
```typescript
const numMatch = sign.address.match(/^(\d+)/);
let forcedDir: 1 | -1 | undefined;
if (numMatch !== null) {
  const oddDir = _streetParity[normalizeToGeometryKey(streetName)];
  if (oddDir !== undefined) {
    const isOdd = parseInt(numMatch[1], 10) % 2 === 1;
    forcedDir = isOdd ? oddDir : (oddDir === 1 ? -1 : 1);
  }
}
waypoints = offsetPolylinePoints(waypoints, sign.lat, sign.lng, LATERAL_OFFSET_M, forcedDir);
```
Use `numMatch` (not `addrMatch`) — `addrMatch` is already declared earlier in both functions for address-range parsing and cannot be re-declared.

## Known gotchas

**Normalization alignment is critical.** The keys written to `street-parity.json` by the build script must match exactly what `normalizeToGeometryKey(streetName)` produces at runtime in `app/map.ts`. `normalizeStreet` in the build script must be a verbatim copy of `normalizeToGeometryKey`:
- `.toUpperCase()`
- `.replace(/\./g, "")` — strip periods
- Abbreviation replacements: STREET→ST, AVENUE→AVE, HIGHWAY→HWY, BOULEVARD→BLVD, PLACE→PL, COURT→CT, DRIVE→DR, ROAD→RD, TERRACE→TER
- Ordinal replacements: FIRST→1ST … SIXTEENTH→16TH (via the same `ORDINAL_TO_NUMERIC` map)
- `.trim()`
- **Do NOT add directional replacements (NORTH→N etc.)** — `normalizeToGeometryKey` does not have them; adding them in the build script would cause divergence.

**`addrMatch` is already declared.** Both `renderTowSegments` and `renderUpcomingTowSegments` already declare `const addrMatch = sign.address.match(/^(\d+)-(\d+)\s+/)`. Do not re-use the name `addrMatch` for the parity extraction — TypeScript strict mode rejects a second `const addrMatch` in the same block. Use `numMatch` with the broader regex `/^(\d+)/` which catches both range and single-number addresses.

**`normalizeToGeometryKey` is not exported** from `map.ts`. The parity lookup calls it directly since both the call sites and the function are in the same file. Do not add an export.

**`forcedDir` skips the `|dot| < 1e-9` early return.** The refactored `offsetPolylinePoints` must branch before computing dot:
```typescript
let dir: 1 | -1;
if (forcedDir !== undefined) {
  dir = forcedDir;
} else {
  // ... compute midLat, midLng, signDY, signDX, dot ...
  if (Math.abs(dot) < 1e-9) return pts;
  dir = dot > 0 ? 1 : -1;
}
```
Without this, a `forcedDir` call on a centerline sign would hit the early return and silently produce no offset — exactly the defect we are fixing.

**User-Agent required for Overpass.** Node's built-in `fetch` sends no User-Agent and may receive HTTP 403 from Overpass. Set `"User-Agent": "hoboken-parking-app/1.0 (build-time street-parity)"` on the request (see F-21 gotcha in CLAUDE.md).

**`isMain` guard** must check both `.ts` and `.js` endings:
```typescript
const isMain =
  typeof process !== "undefined" &&
  Array.isArray(process.argv) &&
  typeof process.argv[1] === "string" &&
  (process.argv[1].endsWith("build-street-parity.ts") ||
    process.argv[1].endsWith("build-street-parity.js"));
```

## Tests

### `offsetPolylinePoints` with `forcedDir` (add to existing describe block)

**Given** a 2-point N-S polyline, sign at midpoint (dot ≈ 0), `forcedDir = 1`,
**When** `offsetPolylinePoints` is called,
**Then** all returned points have shifted east (positive longitude delta) — `forcedDir` overrides the early-return guard.

**Given** same polyline and centerline sign, `forcedDir = -1`,
**When** called,
**Then** all returned points have shifted west (negative longitude delta).

**Given** a 2-point N-S polyline, sign east of center (dot > 0 → natural dir = 1), `forcedDir = -1`,
**When** called,
**Then** all returned points shift west — `forcedDir` overrides the natural GPS direction.

### `renderTowSegments` — address-parity curb offset (new describe block "F-41")

Setup: `initStreetParity({ "BLOOMFIELD ST": 1 })` — odd addresses are on the +1 (left-perpendicular) side.

**Given** a sign with address `"1037-1037 BLOOMFIELD ST"` (odd), road geometry centered on the sign (dot ≈ 0),
**When** `renderTowSegments` is called,
**Then** the inner polyline's points are shifted in the +1 direction (latitude or longitude shifted by > 0 relative to unshifted waypoints).

**Given** a sign with address `"1036-1036 BLOOMFIELD ST"` (even), same setup,
**When** called,
**Then** the inner polyline's points are shifted in the −1 direction.

**Given** `initStreetParity({})` (no entry for any street) and a sign on the centerline,
**When** called,
**Then** the polyline is not rendered (waypoints returned unchanged from dot ≈ 0 path → `pts` reference equality fails `length < 2` check does not apply — graceful degradation: polyline may not render or may coincide with centerline; assert that no crash occurs and the layer count is 0 or matches the fallback behavior).

### `renderUpcomingTowSegments` — mirror tests

Same 3 tests as above, verifying orange (F-35) upcoming segments use the same parity path.

### Normalization alignment (new test)

**Given** `initStreetParity({ "8TH ST": 1 })` and a sign with address `"805 EIGHTH ST"` on a road segment named "EIGHTH ST" in `_roadGeometry`,
**When** `renderTowSegments` is called,
**Then** the inner polyline is offset (streetKey lookup via `normalizeToGeometryKey("EIGHTH ST")` → `"8TH ST"` hits the parity map).
