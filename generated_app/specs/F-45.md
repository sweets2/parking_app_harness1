# F-45 — Fix: Street Cleaning Block-Level Highlight Clipping

## What this is

`renderViolationHighlights` currently groups cleaning entries by `(street, side)`,
ignoring the `location` field entirely. Washington St. has 5 entries across 2 sides
with non-overlapping ranges ("Observer Hwy. to Seventh St.", "Seventh St. to Eighth
St.", etc.). The entire street glows red/orange when only one specific block is
cleaning.

This feature adds block-level clipping so each entry's `location` field constrains
which road-geometry ways are rendered for that entry.

Files modified:

- `app/map.ts` — 4 new helpers + extract `drawWaysHighlight` + refactor `renderViolationHighlights`
- `tests/unit/map.test.ts` — 1 new test case

### Four new private helpers (insert after `normalizeToGeometryKey`, before `mergeWays`)

```typescript
function getMainStreetLng(streetKey: string): number | null {
  const ways = _roadGeometry[streetKey];
  if (ways === undefined || ways.length === 0) return null;
  const lngs: number[] = [];
  for (const way of ways) {
    for (const pt of way) lngs.push(pt[1]);
  }
  if (lngs.length === 0) return null;
  lngs.sort((a, b) => a - b);
  return lngs[Math.floor(lngs.length / 2)];
}

function getCrossStreetLat(crossKey: string, mainLng: number): number | null {
  const ways = _roadGeometry[crossKey];
  if (ways === undefined || ways.length === 0) return null;
  let bestLat: number | null = null;
  let bestDist = Infinity;
  for (const way of ways) {
    for (const pt of way) {
      const dist = Math.abs(pt[1] - mainLng);
      if (dist < bestDist) { bestDist = dist; bestLat = pt[0]; }
    }
  }
  if (bestLat === null || bestDist > 0.005) return null;
  return bestLat;
}

function parseLocationLatBounds(
  location: string,
  streetKey: string,
): [number, number] | null {
  const match = LOCATION_RANGE_RE.exec(location);
  if (match === null) return null;
  const mainLng = getMainStreetLng(streetKey);
  if (mainLng === null) return null;
  const latA = getCrossStreetLat(normalizeToGeometryKey(match[1] ?? ""), mainLng);
  const latB = getCrossStreetLat(normalizeToGeometryKey(match[2] ?? ""), mainLng);
  if (latA === null || latB === null) return null;
  const PAD = 0.0003;
  return [Math.min(latA, latB) - PAD, Math.max(latA, latB) + PAD];
}

function filterWaysByLat(
  ways: [number, number][][],
  minLat: number,
  maxLat: number,
): [number, number][][] {
  return ways.filter(way => {
    if (way.length === 0) return false;
    const lats = way.map(pt => pt[0]);
    return Math.max(...lats) >= minLat && Math.min(...lats) <= maxLat;
  });
}
```

**Helper contracts:**

- `getMainStreetLng(streetKey)` — returns the median longitude of all points across
  all ways for `streetKey`. Used to locate where cross streets should intersect the
  main street. Works correctly for N-S streets (nearly constant longitude). Returns
  null if street is absent from `_roadGeometry`.

- `getCrossStreetLat(crossKey, mainLng)` — scans all ways of `crossKey`, returns the
  lat of the point whose longitude is closest to `mainLng`. Returns null if no point
  is within 0.005° longitude (~400 m) of `mainLng`. This threshold is generous enough
  for Hoboken's grid.

- `parseLocationLatBounds(location, streetKey)` — parses `"Seventh St. to Eighth St."`
  format. Normalises each cross-street name via `normalizeToGeometryKey`, looks up
  cross-street lats, pads each bound by 0.0003° (~33 m), returns `[minLat, maxLat]`.
  Returns null on any failure (unrecognised format, cross street not in geometry, etc.)
  — callers fall back to full-street rendering.

- `filterWaysByLat(ways, minLat, maxLat)` — bounding-box overlap test: includes a way
  if `Math.max(...lats) >= minLat && Math.min(...lats) <= maxLat`. A way is excluded
  only if it lies entirely outside the range. Ways that straddle the boundary appear
  in adjacent block renders; the sub-meter visual overlap is acceptable.

### Refactor `renderViolationHighlights`

Change the grouping key from `(street, side)` to `(street, side, location)`. Replace
the function body entirely:

```typescript
export function renderViolationHighlights(
  cleaningEntries: StreetCleaningEntry[],
  now: Date,
): void {
  _lastCleaningEntries = cleaningEntries;
  _lastNowForHighlights = now;
  clearViolationHighlights();
  if (_map === null) return;

  const blockStatus = new Map<string, "active" | "upcoming">();

  for (const entry of cleaningEntries) {
    const street = normalizeToGeometryKey(entry.street);
    const key = `${street}|||${entry.side}|||${entry.location}`;
    if (isScheduleActiveNow(entry.schedule, now)) {
      blockStatus.set(key, "active");
    } else if (isScheduleUpcomingSoon(entry.schedule, now)) {
      if (blockStatus.get(key) !== "active") {
        blockStatus.set(key, "upcoming");
      }
    }
  }

  for (const [key, status] of blockStatus) {
    const [street = "", side = "Both", location = ""] = key.split("|||");
    const allWays = _roadGeometry[street];
    if (allWays === undefined || allWays.length === 0) continue;

    const bounds = parseLocationLatBounds(location, street);
    const ways = bounds !== null
      ? filterWaysByLat(allWays, bounds[0], bounds[1])
      : allWays;
    if (ways.length === 0) continue;

    const color   = status === "active" ? "#ef4444" : "#f97316";
    const opacity = status === "active" ? 0.28 : 0.22;
    drawWaysHighlight(ways, color, opacity, side !== "Both" ? side : null);
  }
}
```

**Behaviour notes:**

- Active-beats-upcoming is preserved **per block** (same street+side+location key):
  a block that receives both an active and an upcoming entry keeps "active".
- Two entries for the same street+side but different locations now each carry their
  own status independently — no cross-block priority collapse.
- The `allSpecific`/`hasActive`/`hasUpcoming` cross-side logic from the old
  implementation is removed. Side="Both" entries still render full-width (null side);
  all other side values render with the curb offset.
- `_lastCleaningEntries` / `_lastNowForHighlights` state assignments at the top are
  preserved (needed for zoom-change redraws).

## Known limitations

- **E-W main streets:** `getCrossStreetLat` resolves lats, not lngs, so it only helps
  N-S main streets. No block-differentiated entries with E-W mains exist in the current
  120-entry dataset. Falls back to full-street gracefully.
- **"X and Y" format:** One entry uses "Eighth St. and Tenth St." (no "to"). Regex
  doesn't match → null → full-street fallback.
